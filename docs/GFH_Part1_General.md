# ЧАСТЬ I — ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

## 1. Назначение

Внутренний веб- и мобильный инструмент ГФХ для оценки эффективности сотрудников, повышения прозрачности работы и формирования объективного рейтинга персонала.

**Система решает следующие задачи:**
- Проведение периодической оценки сотрудников по гибким критериям
- Формирование рейтинга каждого сотрудника с учётом положительных критериев и антибонусов
- Обеспечение прозрачности — каждый сотрудник видит свой рейтинг и его состав
- Поддержка процедуры оспаривания оценок с доказательной базой
- Детальная иерархическая аналитика и отчётность для всех уровней управления
- Детальный аудит всех действий в системе

## 3. Иерархия и роли

### 3.1 Организационная структура

```
Председатель
└── Заместитель Председателя (блок)
    └── Начальник департамента
        └── Начальник отдела
            └── Сотрудник
```

### 3.2 Роли и видимость данных

| Роль (enum) | Уровень | Что видит в системе |
|---|---|---|
| `CHAIRMAN` | 1 | Всех сотрудников организации, аналитику всех уровней |
| `DEPUTY_CHAIRMAN` | 2 | Свой блок и всё ниже (департаменты, отделы, сотрудники блока) |
| `HEAD_OF_DEPARTMENT` | 3 | Свой департамент и всё ниже (отделы, сотрудники департамента) |
| `HEAD_OF_DEPARTMENT_UNIT` | 4 | Свой отдел и его сотрудников |
| `EMPLOYEE` | 5 | Только свой личный дашборд |
| `ADMIN` | — | Технический аккаунт, не участвует в оценке |

### 3.3 Принцип оценки

- Каждый руководитель оценивает только своих прямых подчинённых
- Рейтинг руководителя состоит из двух компонентов (см. раздел 4.5):
  - Собственный рейтинг по критериям (как у обычного сотрудника)
  - Средний рейтинг его прямых подчинённых
- Пропорции `w1` (собственный) и `w2` (подчинённые) настраиваются администратором, `w1 + w2 = 1`
- Каждый сотрудник видит итоговый рейтинг и детальную разбивку по критериям

### 3.4 Резолв оценщика (resolveEvaluator)

Система автоматически определяет, кто должен оценивать конкретного сотрудника:

```
function resolveEvaluator(userId, periodId):
    # Шаг 1: проверка активного делегирования
    delegation = findActiveDelegation(userId, periodId.start_date)
    if delegation != NULL:
        return delegation.delegated_to_id

    user = getUser(userId)

    # Шаг 2: Председатель — никто не оценивает
    if user.role == CHAIRMAN:
        return NULL

    # Шаг 3: Есть активный прямой руководитель
    if user.manager_id != NULL AND user.manager.is_active:
        return user.manager_id

    # Шаг 4: Руководитель удалён/деактивирован — идём вверх по иерархии
    current = user.manager
    while current != NULL:
        current = current.manager
        if current != NULL AND current.is_active:
            return current.id

    # Шаг 5: Иерархия оборвалась — никто не оценивает
    return NULL
```

**Следствия:**
- Если у сотрудника есть подчинённые, но нет доступного оценщика — итоговый рейтинг = только `avg_subordinate_rating` (как у Председателя)
- Если у сотрудника нет ни подчинённых, ни оценщика — рейтинг не может быть посчитан, период помечается `UNEVALUABLE`, ADMIN уведомляется
- Все случаи применения Шагов 4–5 фиксируются в `audit_log` с пометкой `AUTO_RESOLVED`

### 3.5 Делегирование оценки (ручное)

На время отпуска / болезни / командировки оценщик может быть временно заменён:

- ADMIN через UI создаёт делегирование: кого заменяет, на кого, с какой даты по какую, причина
- Запись хранится в таблице `evaluator_delegations`
- Система при `resolveEvaluator` первым делом проверяет активное делегирование
- Каждое делегирование фиксируется в `audit_log`

### 3.6 Таблица делегирований

```sql
evaluator_delegations (
  id, evaluatee_id, original_evaluator_id, delegated_to_id,
  valid_from, valid_to, reason,
  created_by, created_at, is_active
)
```

## 4. Критерии оценки

### 4.1 Типы критериев по scope

| Scope | Применяется к | Кто управляет |
|---|---|---|
| `GLOBAL` | Все сотрудники | ADMIN |
| `BLOCK` | Сотрудники блока | DEPUTY_CHAIRMAN / ADMIN |
| `DEPARTMENT` | Сотрудники департамента | HEAD_OF_DEPARTMENT / ADMIN |
| `UNIT` | Сотрудники отдела | HEAD_OF_DEPARTMENT_UNIT / ADMIN |
| `INDIVIDUAL` | Конкретный сотрудник | Непосредственный руководитель / ADMIN |

### 4.2 Положительные критерии vs антибонусы

Каждый критерий любого scope имеет булев флаг `is_penalty`:
- `is_penalty = false` — обычный (положительный) критерий, вклад в рейтинг положительный
- `is_penalty = true` — антибонус, вклад в рейтинг отрицательный

**Применение:**
- Антибонусы выставляются руководителем так же, как обычные критерии — через score 0–100 и комментарий
- Score=0 по антибонусу означает «нарушений нет» (штраф 0), score=100 означает «нарушение в полном объёме» (штраф в полный вес критерия)
- Руководитель видит при оценке обычные критерии и антибонусы в разных секциях UI

**Заморозка флага `is_penalty`:**
- После первого использования критерия в оценке (появление записи в `evaluation_scores`) флаг `is_penalty` становится неизменяемым
- Удаление критерия после использования невозможно — только деактивация (`is_active = false`)
- Деактивированный критерий не применяется в новых периодах, но история предыдущих периодов сохраняется

**Автоматические антибонусы:**
- Создаются системой при нарушении дедлайнов (см. раздел 6.1)
- Технически это обычные критерии с `is_penalty=true`, `is_system=true`, с автоматически выставленным score=100
- Не могут быть отредактированы или удалены руководителем

### 4.3 Атрибуты критерия

```
id, name_ru, name_kg, description_ru, description_kg,
scope, scope_entity_id, weight (decimal),
is_penalty BOOLEAN, is_system BOOLEAN, is_active BOOLEAN,
flag_frozen BOOLEAN (становится true при первом использовании),
created_by, created_at, updated_at
```

### 4.4 Логика весов

**Положительные критерии (`is_penalty=false`):**
- ADMIN задаёт суммарный вес GLOBAL-критериев (например, 60%) — хранится в `system_settings`
- Оставшийся пул (40%) распределяется руководителем подразделения между своими критериями
- Критерии наследуются по иерархии: сотрудник отдела получает GLOBAL + BLOCK + DEPARTMENT + UNIT + INDIVIDUAL
- Сумма всех применимых положительных весов для конкретного сотрудника **строго = 100%** — система блокирует сохранение при несоответствии

**Антибонусы (`is_penalty=true`):**
- Идут сверх 100% положительных критериев
- Нет верхнего ограничения на суммарный «анти-вес», но в UI при создании антибонуса руководитель видит текущий суммарный анти-вес для затрагиваемых сотрудников (для осознанного решения)
- Итоговый рейтинг сотрудника может быть отрицательным

### 4.5 Шкала оценки и формулы

**Шкала score по каждому критерию:** 0–100% (одинаково для положительных и антибонусов).

**Итоговый рейтинг:** теоретически от –∞ до 100%, на практике ограничен суммой весов антибонусов. UI отображает с цветовой индикацией:
- Красный: < 40% или отрицательное значение
- Жёлтый: 40–70%
- Зелёный: > 70%

**Рейтинг обычного сотрудника (без подчинённых):**
```
own_criteria_rating = Σ (score_i × weight_i / 100)     // по критериям с is_penalty=false
anti_bonus_penalty  = Σ (score_j × weight_j / 100)     // по критериям с is_penalty=true
final_rating        = own_criteria_rating − anti_bonus_penalty
```

**Рейтинг руководителя (есть подчинённые):**
```
final_rating = (own_criteria_rating × w1) + (avg_subordinate_rating × w2) − anti_bonus_penalty
```
где `avg_subordinate_rating = AVG(final_rating активных подчинённых за тот же период и тот же тип периода)`.

**Рейтинг Председателя (и любого сотрудника без оценщика, но с подчинёнными):**
```
final_rating = avg_subordinate_rating − anti_bonus_penalty
```

**Правила вычисления:**
- Расчёт `avg_subordinate_rating` идёт каскадом снизу вверх по дереву иерархии
- Порядок расчёта: топологическая сортировка дерева от листьев к корню (`RatingService.calculateAllRatings(periodId)`)
- Уволенные сотрудники включаются в `avg_subordinate_rating` только за те периоды, в которые они были активны

### 4.6 Пример расчёта рейтинга

Сотрудник отдела с 5 положительными критериями (суммарный вес 100%) и 2 антибонусами (веса 5% и 3%):

| Критерий | is_penalty | Вес | Score | Вклад |
|---|---|---|---|---|
| GLOBAL-1 | false | 30% | 80 | +24 |
| GLOBAL-2 | false | 30% | 90 | +27 |
| DEPARTMENT-1 | false | 20% | 75 | +15 |
| UNIT-1 | false | 10% | 85 | +8.5 |
| INDIVIDUAL-1 | false | 10% | 70 | +7 |
| UNIT-ANTI-1 (опоздания) | true | 5% | 40 | −2 |
| DEPT-ANTI-1 (нарушение дедлайнов) | true | 3% | 0 | 0 |
| **Итого** | | 100%+8% | | **79.5** |

## 5. Периоды оценки

### 5.1 Типы периодов

| Тип | Частота |
|---|---|
| `MONTHLY` | Ежемесячно |
| `QUARTERLY` | Ежеквартально |
| `SEMI_ANNUAL` | Каждые 6 месяцев |
| `ANNUAL` | Раз в год |

### 5.2 Параллельные периоды

- В системе одновременно могут быть активны периоды разных типов для всех сотрудников
- Каждый сотрудник видит в своём дашборде все свои активные периоды отдельно
- Руководитель видит в `/api/evaluations/to-do` задачи по всем активным периодам, сгруппированные по типу периода
- В `avg_subordinate_rating` для расчёта рейтинга руководителя за период типа X берутся рейтинги подчинённых за тот же тип периода
- Сравнение рейтингов между разными типами периодов невозможно — в аналитике фильтр по типу обязателен

### 5.3 Расписание и дедлайны

- Запуск автоматический (по расписанию типа периода)
- Дедлайн по умолчанию задаётся в `system_settings` для каждого типа отдельно
- Все сроки считаются в часовом поясе Asia/Bishkek
- «Рабочие дни» на реакцию сотрудника считаются с учётом производственного календаря КР
- Напоминания: за 1 день и за 1 час до дедлайна (email + in-app)

## 6. Workflow — полный жизненный цикл оценки

```
[СИСТЕМА] Автозапуск цикла по расписанию → статус: OPEN
    ↓
[СИСТЕМА] resolveEvaluator для каждого сотрудника → создание записей evaluations
    ↓
[РУКОВОДИТЕЛЬ] Выставляет оценки подчинённым по всем применимым критериям
               (положительным и антибонусам): score 0–100 + комментарий + файл
    ↓
[СИСТЕМА] Рассчитывает рейтинг → сотрудник получает уведомление
    ↓
[СОТРУДНИК] Реагирует на каждый критерий (включая антибонусы): AGREE / DISAGREE
            При DISAGREE → текстовое обоснование + файлы-доказательства
    ↓
[СИСТЕМА] Если сотрудник не реагирует в срок → автоматически AGREE + авто-антибонус сотруднику
    ↓
[РУКОВОДИТЕЛЬ] После реакции всех подчинённых → отправляет оценки на утверждение
    ↓
[ВЫШЕСТОЯЩИЙ РУКОВОДИТЕЛЬ] Рассматривает апелляции
                            Решение по каждой: UPHELD / OVERTURNED
                            При OVERTURNED → указывает новый score и комментарий
                            Решение апелляции является окончательным
    ↓
[СИСТЕМА] Пересчёт рейтингов снизу вверх по иерархии → период CLOSED → обновляет аналитику
```

### 6.1 Автоматические антибонусы

| Событие | Последствие |
|---|---|
| Руководитель не выставил оценку до дедлайна | Всем подчинённым ставится `auto_score_default` по всем положительным критериям. Применяется авто-антибонус с весом `auto_anti_bonus_manager_pct` к руководителю и всем руководителям вверх по иерархии. |
| Сотрудник не отреагировал в срок | Автоматическое AGREE по всем критериям. Применяется авто-антибонус с весом `auto_anti_bonus_employee_pct` к сотруднику. |

Автоматические антибонусы применяются как системные критерии с `is_penalty=true`, `is_system=true`, score=100. Действуют только в том периоде, где произошло нарушение.

### 6.2 Статусы периода оценки

```
SCHEDULED → OPEN → EVALUATION_IN_PROGRESS → APPROVAL_IN_PROGRESS → CLOSED
                                                                ↓
                                                          UNEVALUABLE (для отдельных оценок)
```

Статус `UNEVALUABLE` применяется к отдельным оценкам, для которых `resolveEvaluator` не нашёл оценщика и у сотрудника нет подчинённых. Период в целом закрывается в `CLOSED` даже при наличии UNEVALUABLE-записей.

### 6.3 Апелляция

- Апелляция подаётся сотрудником на конкретный критерий (положительный или антибонус, включая автоматический)
- Рассматривает её непосредственный руководитель оценщика (уровнем выше)
- Решение (`UPHELD` / `OVERTURNED`) является окончательным, повторная апелляция невозможна
- При `OVERTURNED` вышестоящий указывает новый score и комментарий
- Если оценщик = Председатель, апелляция недоступна в UI. При несогласии сотрудник может оставить комментарий без процессуальных последствий
- Апелляция на автоматический антибонус рассматривается оценщиком уровнем выше «виновника» (руководителя, пропустившего дедлайн, или самого сотрудника для антибонуса за пропуск реакции)

## 8. Дашборды и аналитика

### 8.1 Личный дашборд (все сотрудники)

- Текущий рейтинг с цветовой индикацией
- Разбивка рейтинга: отдельный блок «Положительные критерии» и отдельный блок «Антибонусы»
- График динамики рейтинга по периодам (line chart), фильтр по типу периода
- Статус текущих циклов оценки (все активные типы периодов)
- Активные апелляции
- Последние уведомления

### 8.2 Иерархическая аналитика

Основной инструмент для всех руководящих ролей. Единая страница с тремя независимыми контролами, через которые пользователь формирует нужный ему вид.

**Три контрола:**

1. **Режим отображения** — как визуализировать данные:
   - Таблица (плоский список с сортировкой и пагинацией)
   - Карточки (плитки с ключевыми показателями)
   - Развёрнутое дерево иерархии
   - График сравнения (bar chart)

2. **Разрез (уровень агрегации)** — по какой группировке показывать:
   - По организации (единый агрегат) — только `CHAIRMAN`
   - По блокам
   - По департаментам
   - По отделам
   - По сотрудникам

3. **Фильтр подразделений** — multi-select конкретных блоков / департаментов / отделов в пределах прав доступа роли. По умолчанию включены все доступные.

**Права доступа к уровням агрегации:**

| Роль | Доступные разрезы |
|---|---|
| `CHAIRMAN` | Организация, блоки, департаменты, отделы, сотрудники |
| `DEPUTY_CHAIRMAN` | Свой блок, департаменты блока, отделы блока, сотрудники блока |
| `HEAD_OF_DEPARTMENT` | Свой департамент, отделы департамента, сотрудники департамента |
| `HEAD_OF_DEPARTMENT_UNIT` | Свой отдел, сотрудники отдела |

**Агрегация:**
- «Итого по блоку / департаменту / отделу / организации» = среднее арифметическое `final_rating` всех активных сотрудников в группе (включая руководителя)
- Агрегат по группе и рейтинг руководителя этой группы — **две разные цифры**, обе отображаются рядом для избежания путаницы («Средний рейтинг отдела: 72%» и «Рейтинг начальника отдела: 78%» в одной строке)
- Уволенные сотрудники включаются в историческую аналитику по умолчанию (есть чекбокс «исключить уволенных»)

**Drill-down:**
- Клик на строку (например, строка отдела в режиме «По отделам») открывает модальное окно с деталями этого отдела: его сотрудники с их рейтингами, разбивка по критериям, список антибонусов
- Модальное окно не меняет основную страницу — пользователь может закрыть окно и продолжить работу с текущим видом

**Сравнительный режим:**
Дополнительный переключатель включает один из трёх режимов сравнения:
- Сравнение подразделений между собой (выбор 2–4 подразделений для параллельного отображения)
- Сравнение текущего периода с предыдущим того же типа (динамика ↑↓)
- Benchmark — текущее подразделение vs средний уровень всей вышестоящей группы (отдел vs средний по департаменту)

**Сохранение настроек:**
- Последние значения трёх контролов и режима сравнения сохраняются в `localStorage` браузера
- При следующем входе аналитика открывается в том же виде, в каком пользователь её оставил

### 8.3 Дашборд руководителя (оперативная сводка)

Отдельная страница для оперативной работы руководителя в текущем цикле оценки (не путать с иерархической аналитикой):

- Прогресс выставления оценок в текущем цикле (X из N), отдельно по каждому типу периода
- Сводная таблица: сотрудник → рейтинг → статус → динамика (↑↓)
- Топ-3 и антитоп-3 сотрудников
- Список активных апелляций на рассмотрении
- Незакрытые задачи («выставить оценку», «рассмотреть апелляцию»)

### 8.4 Аналитика антибонусов (отдельная страница)

Доступна для всех руководящих ролей в пределах их прав. Показывает:

- Суммарный анти-вес, применённый к каждому сотруднику в группе (сортировка по убыванию)
- Топ-10 сотрудников, получивших наибольший `anti_bonus_penalty`
- Распределение антибонусов по критериям (какие нарушения встречаются чаще всего)
- Динамика антибонусов в группе за последние 12 периодов
- Отдельно — статистика автоматических антибонусов (пропуски дедлайнов)
- Фильтры: период, тип антибонуса (ручной / автоматический), подразделение

### 8.5 Экспорт отчётов

| Отчёт | Формат | Кто |
|---|---|---|

| Личный рейтинг с детализацией | PDF | Каждый сотрудник (свой) |
| Сравнение подразделений | Excel + PDF | По правам доступа |
| Динамика рейтингов | Excel + PDF | По правам доступа |
| Статистика апелляций | Excel + PDF | ADMIN, CHAIRMAN |
| Аналитика антибонусов | Excel | По правам доступа |

Все экспорты содержат: логотип ГФХ, дату формирования, имя сформировавшего, параметры фильтра. Факт экспорта фиксируется в `audit_log`.

## 9. Аудит

Детальный append-only журнал. Записи **нельзя изменить или удалить** (PostgreSQL триггер).

**Охват:**
- Аутентификация (вход, выход, неудачные попытки, блокировка, смена пароля, сброс пароля)
- CRUD пользователей и оргструктуры
- CRUD критериев, изменение весов, деактивация критериев
- Все оценки (включая автоматические с пометкой `AUTO`)
- Апелляции и решения по ним
- Антибонусы — ручные (как обычные оценки) и автоматические (с причиной)
- Изменения `system_settings`
- Открытие/закрытие периодов
- Факты экспорта отчётов
- Применение `resolveEvaluator` (Шаги 4–5)
- Создание/отмена делегирований
- Загрузка и скачивание файлов

**Структура записи:**
```
id, timestamp, user_id, user_name, action, entity_type, entity_id,
old_value (JSONB), new_value (JSONB), ip_address, user_agent
```

Просмотр — только `ADMIN` и `CHAIRMAN`. Экспорт в Excel — только `ADMIN`.

## 10. Безопасность

### 10.1 Аутентификация и пароли

**Политика паролей:**
- Минимум 10 символов
- Минимум одна заглавная буква, одна строчная, одна цифра, один спецсимвол
- Проверка по словарю топ-10 000 самых популярных паролей
- Срок действия пароля: 90 дней, обязательная смена при истечении
- История: нельзя повторить последние 5 паролей
- Хранение: bcrypt (cost factor ≥ 12) или argon2id

**Защита от brute force:**
- Rate limiting на `/api/auth/login`: не более 5 попыток за 15 минут с одного IP
- Блокировка аккаунта на 30 минут после 5 неудачных попыток подряд
- Уведомление сотрудника и ADMIN о блокировке (email)

**Восстановление пароля:**
- Самостоятельный сброс через email (ссылка с одноразовым токеном, срок жизни 30 минут)
- ADMIN может принудительно сбросить пароль (выдаётся временный одноразовый пароль, требующий смены при первом входе)

**Токены:**
- Access token: JWT, срок жизни 15 минут
- Refresh token: срок жизни 7 дней, ротация при каждом использовании
- При повторном использовании refresh token — все сессии пользователя инвалидируются
- Refresh token хранится в таблице `refresh_tokens` с возможностью отзыва ADMIN-ом

**HTTPS:**
- Вся система работает только по HTTPS, даже во внутренней сети (TLS 1.2+)
- HSTS включён

### 10.2 Безопасность файлов

**Проверки при загрузке:**
- MIME-sniffing (проверка реального типа по magic bytes)
- Whitelist разрешённых типов из `system_settings.allowed_file_types`
- Санитация имени файла (удаление `../`, `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`, NULL-байтов; длина ≤ 255 символов)
- Оригинальное имя сохраняется в `original_name`, на диске хранится под UUID
- Проверка размера: ≤ `max_file_size_mb`

**Хранение:**
- Файлы вне корня веб-сервера
- Структура директорий: `/var/lib/gfh/files/{year}/{month}/{uuid}.{ext}`
- Права на файловой системе: `chmod 600` для пользователя приложения

**Доступ (защита от IDOR):**
- Скачивание только через API endpoint с проверкой прав
- Доступ имеют только: (а) загрузивший, (б) участники конкретной оценки (оценщик, оцениваемый, вышестоящий при апелляции), (в) ADMIN, (г) CHAIRMAN
- Каждое скачивание фиксируется в `audit_log`

### 10.3 Защита от общих уязвимостей

- **SQL injection:** только параметризованные запросы / JPA
- **XSS:** экранирование вывода, валидация всех пользовательских строк на бэкенде
- **CSRF:** JWT в `Authorization` header; на форме сброса пароля — CSRF-токен
- **IDOR:** проверка владения ресурсом на каждом endpoint согласно иерархии
- **Логирование чувствительных данных:** пароли, токены, содержимое оценок не попадают в логи приложения

## 12. Персональные данные и хранение

- Обработка ПДн соответствует требованиям Закона КР «О персональных данных»
- При первом входе сотрудник соглашается с политикой обработки ПДн (чек-бокс, факт согласия фиксируется в `pdpa_consents`)
- Сотрудник имеет право на выгрузку всех своих данных (endpoint `/api/me/export`, формат JSON + PDF)
- **Уволенные сотрудники:** данные хранятся бессрочно, учитываются в исторической аналитике
- При деактивации пользователя: доступ в систему блокируется немедленно; все активные refresh tokens отзываются; исторические оценки сохраняются
- Удаление данных из системы — только через audit_log-сопровождаемую процедуру по письменному запросу (не реализуется в v1)

## 13. Эксплуатация (on-premise)

### 13.1 Бэкапы
- PostgreSQL: `pg_dump` ежедневно в 03:00 Asia/Bishkek
- WAL-архивирование для Point-in-Time Recovery
- Файловое хранилище: rsync на резервный сервер ежедневно в 04:00
- Срок хранения бэкапов: 30 дней ежедневных + 12 месяцев ежемесячных
- Ежеквартальная проверка восстанавливаемости

### 13.2 Мониторинг и логирование
- Логи приложения: Logback, ротация по дням, хранение 90 дней
- Уровень логирования настраивается через `application.properties` (INFO по умолчанию)
- Health-check endpoint: `/actuator/health`
- Метрики: Spring Boot Actuator (`/actuator/metrics`)

### 13.3 Первичный запуск
- Первичное создание суперадмина через миграцию Liquibase (временный пароль в `.env`, обязательная смена при первом входе)
- ADMIN вручную через UI создаёт всех ~100 сотрудников и оргструктуру
- Первый цикл оценки рекомендуется запустить в тестовом режиме (флаг `dry_run` в настройках периода)
